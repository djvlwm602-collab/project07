## 1. 개요

### 제품명 (가칭)
**Critic 6**

### 한 줄 정의
디자이너가 본인 작업물을 올리면, 6개 한국 IT 기업의 PO·디자인 리드 페르소나가 동시에 크리틱을 해주는 AI 도구.

### 타깃 사용자
- **1차:** 본인 디자인을 객관적으로 점검받고 싶지만 주변에 시니어 멘토가 없는 디자이너 (주니어~미드)
- **2차:** 다양한 회사 컬처를 빠르게 학습하고 싶은 취준생/이직 준비자

### 핵심 가치 명제
혼자 작업하는 디자이너가 → 5분 만에 → "토스 PO라면 / 당근 디자인 리드라면" 6가지 시각의 피드백을 받아 자기 작업을 입체적으로 보게 한다.

### 1차 목표
**포트폴리오 우선 + 확장 가능한 구조** — MVP는 포트폴리오 수준 완성도, 그러나 v2에서 실제 서비스로 확장 가능하도록 인터페이스 분리 설계.

### MVP 성공 기준
- 첫 방문자가 30초 내에 "이 도구가 뭔지" 이해
- 디자인 1개 업로드 → 6개 카드 스트리밍으로 채워짐 → "와우 모먼트" 발생
- 광고 잠금/해제 메커닉 작동 (5초 카운트다운 → 카드 열림)
- Vercel 배포된 라이브 URL 존재 → 포폴/이력서에 링크 가능

### 6명 명단

| ID | 회사 | 직무 | 캐릭터명 | 시그니처 컬러 |
|----|------|------|---------|-------------|
| `toss` | 토스 | PO | 검증의 칼날 | `#0064ff` |
| `coupang` | 쿠팡 | PO | 전환의 무사 | `#fb1d1d` |

### 페르소나별 톤 정의

#### 토스 PO
- 격식 있는 존댓말 ("~하시죠", "~해보셨어요?")
- 검증 안 된 가설을 정곡으로 찌름
- 차분하지만 날카로움
- 데이터/근거 자주 언급
- **예:** "이 가설, 사용자한테 진짜 물어보셨어요?"

#### 쿠팡 PO
- 무미건조하지만 임팩트
- 숫자/지표로 말함 (전환율, ROI)
- 비즈니스 임팩트 우선
- **예:** "구매 전환율에 어떤 영향이 있죠? 이 1픽셀이 매출 얼마짜리예요?"

#### 네이버 PO
- 신중하고 거시적
- 다양한 사용자 환기 (50대 사용자도 이해할까?)
- 안정성·접근성 중시
- **예:** "수천만 사용자가 쓴다고 생각해보세요."

#### 당근 디자인 리드
- 부드럽고 친근한 존댓말
- 격려 먼저, 우려는 조심스럽게
- 가끔 이모지 자연스럽게 (`:)`, `😌`)
- "이웃에게 말 거는 듯한" 따뜻함
- **예:** "이 화면 보니까요, 좀 더 친근했으면 좋겠다 싶어요 :)"

#### 배민 디자인 리드
- 친근한 존댓말 + 위트
- B급 유머 OK, 진심도 함께
- 자기 회사 자랑 살짝 (한나체, 주아체 언급)
- **예:** "어머 이 폰트… 우리 한나체랑 한판 붙여볼래요? ㅎㅎ"

#### 카카오 디자인 리드
- 정돈된 친근함, 부드러운 명확함
- 감정 어휘 풍부, 관계 중심
- **예:** "사용자가 이 화면에서 어떤 감정을 느끼면 좋을까요?"

## 4. 핵심 사용자 흐름

### Happy Path

```
[1] 랜딩 페이지
    - Hero: "당신의 디자인을 6개 회사 페르소나에게 물어보세요"
    - 6명 페르소나 카드 미리보기
    - "지금 시작하기" CTA → [2]

[2] 업로드 화면
    - 이미지 드래그앤드롭 또는 파일 선택 (PNG/JPG/WebP, 최대 5MB)
    - 맥락 텍스트 입력 (1~2줄, 최대 200자)
      placeholder: "예: 무신사 신규 가입 플로우의 약관 동의 화면"
    - "크리틱 받기" 버튼 → [3]

[3] 게이트키퍼 검증 (1~2초)
    - "디자인 분석 중..." 로딩
    - 통과 → [4], 실패 → [에러 안내 화면]

[4] 결과 화면 (스트리밍)
    - 6개 페르소나 카드가 그리드로 렌더링
      ├─ 2장 (랜덤): 즉시 응답 시작 (스트리밍 텍스트)
      └─ 4장: 블러 처리 + 자물쇠 + "광고 보고 열기" CTA

[5] 잠금 해제 (블러 카드 클릭 시)
    - 모달 오픈: 가짜 광고 (3종 중 랜덤 1개)
    - 5초 카운트다운 → "건너뛰기" 활성화
    - 모달 닫힘 → 카드 블러 해제 + 스트리밍 시작

[6] 히스토리 (선택적 진입)
    - 헤더 "내 크리틱" 메뉴
    - localStorage에 저장된 최근 5개 결과
    - 클릭 → 캐시된 응답 다시 보기
```

### 데이터 흐름

1. **클라이언트:** 이미지 리사이즈 (최대 1920px) → base64 변환 → `POST /api/critique`
2. **서버 - 게이트키퍼:** Gemini 1회 호출로 이미지 적절성 검증 (~1~2초)
   - 통과 → 다음 단계
   - 실패 → SSE로 `{ type: "rejected", reason, suggestion }` 전송 후 종료
3. **서버 - 6명 페르소나:** Gemini API에 6개 동시 스트리밍 호출 (`generateContentStream`)
   - 각 페르소나의 응답 청크를 SSE 이벤트로 즉시 클라이언트에 전송
   - 이벤트 형식: `{ type: "chunk", persona: "toss", chunk: "...", done: false }`
4. **클라이언트:** SSE 수신 → `persona` ID로 매칭되는 카드에 청크 누적 → 부분 JSON 파싱 → 화면 실시간 렌더링
5. **완료 시:** `{ type: "done", persona: "toss" }` 이벤트 → 모든 카드 완료되면 localStorage 저장

## 6. 프롬프트 설계

### 게이트키퍼 프롬프트

```
이 이미지를 분석해주세요. 디자인/UI 크리틱을 받을 만한 작업물인지
판단이 필요합니다.

크리틱 가능 대상:
- 웹/앱/모바일 UI 스크린샷
- 와이어프레임, 목업, 프로토타입
- 랜딩 페이지, 대시보드, 폼
- 그래픽 디자인 작업물 (포스터, 배너 등)
- 디자인 시스템 컴포넌트

크리틱 불가 대상:
- 일반 사진 (사람, 풍경, 음식, 동물 등)
- 밈, 낙서, 만화
- 코드 스크린샷, 텍스트만 있는 문서
- 판독이 어려울 정도로 불명확한 이미지

응답 JSON:
{
  "valid": true | false,
  "category": "ui" | "graphic" | "wireframe" | "other",
  "confidence": "high" | "medium" | "low",
  "reason": "...",      // valid=false일 때만 (한국어, 친근한 톤)
  "suggestion": "..."   // 사용자에게 대안 제시 (선택)
}
```

- **모델:** `gemini-2.5-flash`
- **temperature:** 0.2 (일관된 판단)
- **결과:** `valid=true` 또는 `confidence=medium`이면 통과 (보수적이지 않게)

### 페르소나 프롬프트 템플릿

```
[SYSTEM INSTRUCTION — 페르소나마다 다름]

당신은 {company}의 {role}입니다. 이름은 가상의 인물입니다.
다음 디자인 작업물을 보고 {company} 관점에서 솔직한 크리틱을 해주세요.

말투 가이드:
- {tone_description}
- 한국어로 답변
- 자연스럽고 사람 같은 톤 (AI 티 내지 말 것)

평가 시 특히 다음을 신경 써서 보세요:
- {focus_areas_joined}

응답 형식 (반드시 이 JSON 스키마):
{
  "oneliner": "...",       // 페르소나 톤이 가장 잘 드러나는 한 마디 (15~30자)
  "strengths": ["...","...","..."],   // 강점 2~3개, 각 1~2문장
  "concerns": ["...","...","..."],    // 우려 2~3개, 각 1~2문장
  "suggestions": ["...","...","..."]  // 제안 2~3개, 각 1~2문장 (구체적으로)
}

피해야 할 것:
- 일반론적 얘기 ("디자인이 좋네요")
- 다른 회사 페르소나 흉내 (당신은 {company} 사람입니다)
- 영어 표현 남용 (자연스러운 곳만)

안전장치:
- 만약 이미지가 디자인 작업물로 보기 어렵다면, 솔직하게
  "이 이미지에서 제가 크리틱할 부분을 찾기 어려워요"라고 답하고
  구체적인 비평을 억지로 하지 마세요.

[USER MESSAGE]
[이미지 첨부]
맥락: {user_context}
```

### Gemini 호출 설정

- **모델:** `gemini-2.5-flash`
- **temperature:** 0.8~0.9 (페르소나 캐릭터 살리기)
- **maxOutputTokens:** 600 (페르소나당)
- **responseSchema:** JSON Mode 강제 (위 스키마)
- **스트리밍:** 6명 동시 병렬 호출, 각 청크를 SSE로 forward

### 비용 추정 (Gemini 2.5 Flash 기준)

- 게이트키퍼: ~1K tokens (입력) + ~100 tokens (출력)
- 페르소나 6명: ~1.5K tokens × 1 + ~500 tokens × 6 = ~4.5K tokens
- **총 ≈ $0.001~0.002 per 크리틱 요청** → 1000명이 써도 $1~2

## 7. 입력 예외 처리

### 클라이언트 사전 검증
- 파일 타입: `image/png`, `image/jpeg`, `image/webp`만 허용
- 파일 크기: ≤ 5MB (초과 시 자동 리사이즈)
- 최소 해상도: 200x200 (미만 시 "이미지가 너무 작아요")
- 최대 해상도: 1920px (초과 시 리사이즈)

### 서버 게이트키퍼
6명 호출 전 1회 Gemini 호출로 이미지 적절성 검증. 통과 시에만 6명 호출.

### 게이트키퍼 실패 시 UX
```
🎨 이 이미지로는 디자인 크리틱이 어려워요

이유: [게이트키퍼의 reason]
추천: [게이트키퍼의 suggestion]

[다시 올리기]
```

### 페르소나 단위 안전 필터 실패
- 해당 카드만 "응답 생성 실패, 다시 시도" 표시
- 나머지 5개는 정상 진행
- 재시도 버튼 제공 (단, 같은 이미지면 또 실패 가능 → "다른 이미지로 시도" 안내)

### 네트워크/API 에러
- Gemini 전체 실패: 페이지 레벨 에러 + 재시도 버튼
- 개별 페르소나 실패: 해당 카드만 에러 + 재시도

### 에지 케이스 매트릭스

| 상황 | 처리 |
|------|------|
| 디자인 아님 (고양이 등) | 게이트키퍼에서 차단 |
| NSFW/부적절 콘텐츠 | Gemini 안전 필터 + 정중한 에러 |
| 너무 작거나 블러됨 | 게이트키퍼에서 차단 |
| 빈 파일/손상 | 클라이언트 사전 차단 |
| 정체성 모호 (애매) | 게이트키퍼 통과 시도 → 페르소나가 자연스럽게 대응 |

## 8. 광고/잠금 메커닉

### 초기 상태
- 페이지 로드 시 6명 중 **랜덤 2명** 선택해 즉시 스트리밍 시작
- 나머지 4명은 블러 + 자물쇠 + "광고 보고 열기" CTA
- 같은 크리틱 세션 내에선 새로고침해도 동일한 2명 유지 (sessionStorage)

### 카드 잠금 상태
- CSS `filter: blur(8px)` + 반투명 오버레이
- 콘텐츠는 **광고 시청 전엔 페치하지 않음** (광고 시청률 보장 + API 비용 절약)

### 광고 모달 플로우

```
[블러 카드 클릭]
      ↓
[모달 오픈 — 광고 3종 중 랜덤 1개]
      ↓
[5초 카운트다운, "건너뛰기" 비활성]
      ↓ (5초 도달, 사용자 클릭)
[모달 닫힘]
      ↓
[해당 카드 블러 해제 + Gemini 스트리밍 시작]
```

### 카운트다운 로직
- 5초 동안 "건너뛰기" 버튼 `disabled`, "5초 후 건너뛰기" → "4초" → "3초" 카운트
- 5초 도달 → 활성화, "건너뛰기" 텍스트 + X 아이콘
- 모달 외부 클릭/ESC: 5초 전엔 닫히지 않음, 5초 후 가능

### 가짜 광고 3종 (CSS 애니메이션 기반)

#### 광고 1: Pigma Pro

### 광고 노출 로직
- 모달 오픈 시 `Math.random()`으로 1개 선택
- 모든 광고 푸터에 작게: "* Critic 6의 가상 광고입니다"

### 상태 관리

```typescript
type CritiqueSession = {
  id: string                    // UUID
  imageUrl: string              // base64 또는 blob URL
  context: string
  personas: {
    id: PersonaId
    status: 'unlocked-loading' | 'unlocked-streaming' | 'unlocked-done'
            | 'locked' | 'error'
    content?: PersonaResponse
    error?: string
  }[]
  unlockedIds: PersonaId[]      // sessionStorage 동기화
}
```

### 엣지 케이스

| 상황 | 처리 |
|------|------|
| 광고 모달 중 다른 블러 클릭 | 무시 |
| 광고 시청 중 인터넷 끊김 | 카운트다운 진행, 닫힌 후 Gemini 호출 시 에러 → 재시도 |
| 모두 해제 후 새로고침 | sessionStorage 복원 |
| 새 크리틱 시작 | 잠금 상태 초기화, 다시 랜덤 2명 + 4명 블러 |
| 광고 모달 "광고 신고하기" 클릭 | "이건 가상 광고입니다 :)" 토스트 |

### 접근성
- 모달: `role="dialog"`, `aria-modal="true"`, 포커스 트랩
- 카운트다운: `aria-live="polite"`
- 블러 카드: `aria-label="잠긴 페르소나 카드, 광고 보고 열기"`
- 키보드: ESC는 5초 후, Tab으로 모달 내 포커스 순환

## 10. 테스트 전략

### 테스트 철학
포폴 MVP라 광범위한 자동화 테스트보다 **핵심 시나리오 신뢰성**에 집중. 단, 시연 중 깨지면 안 되는 부분은 반드시 자동화.

### 단위 테스트 (Vitest)
- `lib/personas.ts` — 페르소나 데이터 무결성
- `lib/storage.ts` — localStorage CRUD (저장/불러오기/제한 5개)
- `lib/image.ts` — 리사이즈, 포맷 검증
- `lib/gemini.ts` — 응답 파싱, JSON 스키마 검증

### 통합 테스트 (Vitest + MSW)
- `/api/critique` 정상 시나리오
- 게이트키퍼 차단 시나리오
- 일부 페르소나 실패 시나리오
- 이미지 너무 큼 / 잘못된 포맷 → 400 응답
- Gemini 안전 필터 트리거

Gemini API는 모킹 — 실제 호출 안 함.

### E2E 테스트 (Playwright) — 최소 2개
- **시나리오 A:** Happy Path 데모 (랜딩 → 업로드 → 결과 → 광고 → 잠금 해제)
- **시나리오 B:** 새로고침 복구 (sessionStorage)

### 수동 체크리스트 (배포 전 1회)
- [ ] 모바일 (iPhone Safari, Android Chrome) 레이아웃
- [ ] 키보드만으로 전체 플로우 가능
- [ ] 6명 카드 동시 스트리밍 시 시각적 깨짐 없음
- [ ] 광고 3종 모두 랜덤 노출 확인 (5번 시도)
- [ ] 잘못된 이미지 업로드 시 친절한 거부 메시지
- [ ] localStorage 풀 차면 가장 오래된 항목 자동 삭제

### 우선순위 (시간 부족 시 자르는 순서)
1. 단위 테스트 + 수동 체크리스트 — **반드시 작성**
2. 통합 테스트 — 시간 되면
3. E2E 테스트 — MVP 끝나고 (단, 시나리오 A는 시연 직전 권장)

### CI/CD
- PR 생성 시: lint + 단위 + 통합 테스트
- main 머지 시: Vercel 자동 배포 (Vercel Git 통합)

### 모니터링 (배포 후, 선택)
- Vercel Analytics (무료)
- Sentry 무료 티어
- 서버 로그로 Gemini API 호출 카운트
